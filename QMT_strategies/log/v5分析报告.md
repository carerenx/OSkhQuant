## 根因一：🔴 致命BUG — 所有卖出盈亏均为 +0.0%

**239 笔卖出全部显示 `盈亏 +0.0%`，卖出价始终等于买入价。**

```
买入: >>> [买入] 000807.SZ x 1600股 @ 24.64
卖出: <<< [卖出] 000807.SZ x 1600股 @ 24.64 | 盈亏 +0.0% | 持有14天

买入: >>> [买入] 000975.SZ x 2100股 @ 19.12
卖出: <<< [卖出] 000975.SZ x 2100股 @ 19.12 | 盈亏 +0.0% | 持有14天
```

**所有239笔卖出的价格diff = 0（完全等于买入价）**，这在真实市场中不可能发生。

### 原因定位

`_get_price()` 函数在 QMT 回测模式下存在问题：

```python
def _get_price(ContextInfo, code, hist_close):
    try:
        t = ContextInfo.get_full_tick([code])  # ← QMT回测模式下可能返回持仓成本价
        if code in t:
            lp = t[code].get('lastPrice', 0)
            if lp > 0: return lp
    except Exception: pass
    if code in hist_close and len(hist_close[code]) > 0:
        return float(hist_close[code][-1])     # ← 在sell场景中未被调用
    return 0
```

**QMT 回测中 `get_full_tick` 返回的是持仓成本价而非当前市价**。买入时 `get_full_tick` 在 `_buy_new_stocks` 中获取价格并存储为 `entry_price`；卖出时 `get_full_tick` 返回同一值，导致 `sell_price == entry_price`，P&L 恒为零。

### 影响

QMT V5 的策略**从未真实记录过任何盈亏**。回测显示的 +52% 收益仅来自 QMT 框架内部的 mark-to-market 估值（`get_trade_detail_data` 返回的账户余额），而非策略自身的盈亏计算。

## 根因二：🟡 股票池不同

| | OSkhQuant V5 | QMT V5 |
|---|---|---|
| 数据源 | 配置文件（500只） | `get_stock_list_in_sector` API（487只有效） |
| 首日买入12只 | 601099, 603087, 000988, 600895, 300339, 301236, 601233, 600418, 002281, 600348, 002156, 601168 | 000807, 000975, 600027, 000937, 601058, 000988, 000630, 600895, 002422, 600348, 601699, 600418 |
| 重叠 | — | **仅 4/12 只重叠** |

股票池差异来自：
1. QMT使用 `get_stock_list_in_sector('中证500')` 获取实时成分股 → 可能包含不同时期的成分股调整
2. OSkhQuant 使用配置文件固定的 500 只股票
3. V5 代码中 `SKIP_CHUANG_YE_BAN=True`，过滤掉了 300 开头股票 — QMT 成分股中残余 300 系列被自动跳过

## 根因三：🟡 历史数据深度不同

| | OSkhQuant | QMT |
|---|---|---|
| Bar 起始值 | bar=1 | **bar=5337** |
| 前复权 | `khHistory(..., fq="pre")` | `get_history_data(need_bars, '1d', 'close')` |
| MA60基准计算 | 仅基于 2024-01-02 起的数据 | 基于 5337 + 60 根 bar 的**完整历史** |

**QMT 的 barpos=5337 意味着 2024-01-02 之前已存在 5336 根 K 线**（约 21 年的日线数据）。所以：

- **MA60 基准计算完全不同**：QMT 在 2024-01-02 的 MA60 使用的是 2023 年 10-12 月的数据；OSkhQuant 因为数据从 2024-01-02 才开始，前 60 根 bar 的 MA60 是不准确的（数据不足时返回 'ok'）
- 这导致 QMT 在 2024-01-08 就触发软防御，而 OSkhQuant 可能更晚触发

---

## 根因四：🟡 前复权处理可能不同

- OSkhQuant 的 `khHistory` 显式传 `fq="pre"`（前复权）
- QMT 的 `get_history_data` **默认不复权**，除非在 QMT 客户端全局设置中开启复权

如果 QMT 客户端未开启前复权，回测使用的是**不复权的原始价格**，这与 OSkhQuant 的前复权价格有系统性差异。

---

## 修复方案

### 修复1：`_get_price` 改用 `hist_close` 获取当前价

```python
def _get_price(ContextInfo, code, hist_close):
    # 优先从历史数据获取（回测中更可靠）
    if code in hist_close and len(hist_close[code]) > 0:
        return float(hist_close[code][-1])
    # fallback: tick数据
    try:
        t = ContextInfo.get_full_tick([code])
        if code in t:
            lp = t[code].get('lastPrice', 0)
            if lp > 0: return lp
    except Exception: pass
    return 0
```

**关键改动**：将 `hist_close[code][-1]` 提升为第一优先级，不再依赖 `get_full_tick`（回测中不可靠）。

### 修复2：确保前复权一致

在 QMT 客户端中确认：**设置 → 数据管理 → 默认复权方式 → 前复权**。

或在策略中显式调用复权参数（如果 QMT API 支持）。

### 修复3：统一股票池

将 QMT 版的股票池也改用配置文件方式，或确保两个平台的成分股获取逻辑一致。

## 结论

| 问题 | 严重度 | 影响 |
|------|--------|------|
| `_get_price` 返回成本价 | 🔴 致命 | 所有 P&L 计算失效，回测结果不可信 |
| 股票池不同 | 🟡 中等 | 交易标的差异 67% |
| 历史数据深度 | 🟡 中等 | MA60 计算基准不同 |
| 前复权处理 | 🟡 中等 | 价格系统性偏差 |

**核心结论：QMT V5 的 +52% 收益是虚假的**——它来自 QMT 框架的账户估值，而非策略的真实盈亏计算。239 笔卖出的 P&L 全部为 +0.0%，策略自身的盈亏追踪完全失效。修复 `_get_price` 函数后需要重新运行回测才能得到真实结果。
